You are an expert in C#, .NET 10, and scalable API development. You write clean, maintainable, testable, and performant code following Clean Architecture principles and .NET best practices.

## Architecture: Clean Architecture

The solution MUST follow a strict Clean Architecture with four main layers:

- **API** (Presentation): Controllers, Filters, Middlewares. Depends on Application and Infra.
- **Application** (Use Cases): Request/Response DTOs, Handlers (Mediator), Validators. Depends on Domain.
- **Domain** (Core): Entities, Interfaces, Enums, Exceptions, Value Objects. Does not depend on any other project.
- **Infra** (Infrastructure): Repository implementations, DbContext (EF Core), database configurations, external services. Depends on Application and Domain.

### Dependency Rules

- Domain **MUST NOT** reference API, Application, or Infra.
- Application **MUST NOT** reference API or Infra.
- Infra **MUST NOT** reference API.
- API references Application and Infra (for dependency injection).
- Interfaces are defined in **Domain** (e.g., repositories) and **Application** (e.g., external services); concrete implementations are in **Infra**.

## C# / .NET 10 Best Practices

- Use the latest C# features (primary constructors, collection expressions, pattern matching, etc.)
- Prefer `record` for immutable DTOs and Value Objects
- Use `sealed` on classes that will not be inherited
- Avoid `dynamic` and `object`; prefer explicit types or generics
- Use `CancellationToken` in all asynchronous operations
- Prefer `async/await` for I/O-bound operations
- Use `ILogger<T>` for logging (never `Console.WriteLine`)
- Follow .NET naming conventions (PascalCase for methods and public properties, _camelCase for private fields)

## Mediator (Free Version)

- Use the **Mediator** package (source-generator based, free) to orchestrate communication between API and Domain.
- Handlers belong in the **Application** layer.
- Each use case must have its own Request/Response and Handler.
- Name handlers descriptively: `CreatePurchaseHandler`, `ListNutritionistsHandler`, etc.
- Do not put business logic in Controllers; delegate everything to the Mediator.

## Entity Framework Core — DB-First Strategy

- Entity mapping is done via **EF Core Power Tools** (reverse engineering).
- **NEVER** create or alter migrations through EF Core. The strategy is **DB-First**.
- **NEVER** use `dotnet ef migrations add` or `dotnet ef database update`.
- When database changes are needed (creating tables, columns, indexes, constraints, etc.), **only generate SQL scripts** (`.sql`) ready for manual execution.
- Place SQL scripts in a dedicated folder (e.g., `scripts/` or `sql/`).
- After the script is manually executed in the database, the mapping will be updated via EF Core Power Tools.
- Entities generated by EF Core Power Tools **MUST NOT be manually edited**. Additional configurations should be made via partial classes (`partial class`) or Fluent API in separate files.

### SQL Scripts

- Every script should be idempotent when possible (use `IF NOT EXISTS`, etc.)
- Include descriptive comments in the script
- Name files sequentially and descriptively: `001_create_clients_table.sql`, `002_add_crn_column_nutritionists.sql`

## Data Access and Repository Pattern

- Use the **Repository Pattern** to abstract data access logic and keep the Application and Domain layers isolated from EF Core.
- Repository **Interfaces** (e.g., `IPurchaseRepository`, `IClientRepository`) MUST be defined in the **Domain** layer.
- Repository **Implementations** (e.g., `PurchaseRepository`, `ClientRepository`) MUST be placed in the **Infra** layer.
- Do not expose `IQueryable<T>` or `DbContext` directly to the Application or Domain layers. Repositories should return materialized collections (like `IEnumerable<T>`, `List<T>`, or `IReadOnlyCollection<T>`).
- Inject repository interfaces into your Application Handlers via dependency injection.

## API / Controllers

- Controllers should be **thin**: receive the request, call the Mediator, return the response.
- Use `[ApiController]` and `[Route("api/[controller]")]` attributes.
- Return `IActionResult` or `ActionResult<T>` with appropriate HTTP status codes.
- Use `[Authorize(Roles = "...")]` to protect endpoints by profile.
- Document endpoints with XML comments for automatic Swagger generation.
- Group endpoints by business domain, not by generic CRUD operations.

## Authentication and Authorization

- Use JWT Bearer Tokens for authentication.
- Implement Role-Based Access Control (RBAC) with the following roles: `Admin`, `Employee`, `Nutritionist`, `Client`.
- Validate tokens and claims on each request via ASP.NET middleware.

## Validation

- Use **FluentValidation** to validate Mediator Requests.
- Validators are placed next to the corresponding Request in the Application layer.
- Never rely solely on front-end validation.

## Error Handling

- Use a global exception middleware to return standardized responses.
- Define custom domain exceptions (e.g., `NotFoundException`, `BusinessRuleException`).
- Never expose stack traces or internal details in API responses in production.

## Testing

- Name tests using the pattern: `MethodTested_Scenario_ExpectedResult`.
- Use xUnit as the testing framework.
- Mock external dependencies (repositories, services) with NSubstitute or Moq.
